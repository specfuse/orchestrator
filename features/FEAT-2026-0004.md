## Description

Feature used by the Phase 2 walkthrough (WU 2.7) to exercise the PM agent v1 end-to-end on a **happy-path, two-repo shape**. Adds a widget-listing capability that spans the persistence component (Python, `orchestrator-persistence-sample`) and the API component (.NET, `orchestrator-api-sample`): a new repository method to list widgets filtered by a quantity floor, and a new HTTP endpoint that surfaces it.

The feature is deliberately shaped to match the worked example in [`agents/pm/skills/task-decomposition/SKILL.md`](../agents/pm/skills/task-decomposition/SKILL.md) §"Worked example": two implementation tasks (one per repo), one `qa_authoring`, one `qa_execution`, one `qa_curation`. No sensitive-path keywords, no cross-feature dependencies, no ambiguous spec shapes. Its purpose is to produce a signal on whether the PM agent can take an approved task graph from `planning` through `plan_review` and into `in_progress` with zero plan edits, all skills cooperating cleanly.

## Behaviors

Each heading below is one user-facing or system-facing behavior the feature delivers. The PM agent's task-decomposition skill (§Step 2, "Feature narrative in `/product/features/*.md` without a formal spec — one capability per explicit 'behavior' or 'user story' heading") consumes them as the capability list.

### Behavior 1 — Persistence port for quantity-filtered listing

The persistence component exposes an async method `list_by_quantity_above_async(threshold: int) -> list[Widget]` on `WidgetRepository` (Protocol) and its `InMemoryWidgetRepository` adapter. The method returns every stored widget whose `quantity` is strictly greater than `threshold`, in insertion order. A threshold of `0` returns every widget with a positive quantity (equivalent to filtering out zero-quantity items).

### Behavior 2 — API endpoint for quantity-filtered listing

The API component exposes `GET /widgets?min_quantity=N` returning a JSON array of widgets whose `quantity` is strictly greater than `N`. `N` is a non-negative integer query parameter; blank or missing `min_quantity` defaults to `0` (all positive-quantity widgets). The handler delegates to the persistence port via the sample repo's existing repository abstraction pattern — the two repos are validated independently in this walkthrough, so the API-side implementation mocks the persistence call rather than making a cross-process request.

## Scope

- Persistence repo: new port method and in-memory adapter method per Behavior 1. Asynchronous, no locking beyond the existing `asyncio.Lock` on the adapter.
- API repo: new endpoint per Behavior 2, handler + request validation + tests for each status code (`200`, `400` on non-integer / negative `min_quantity`).
- QA: one authored test plan covering both behaviors, one execution run, one curation pass to decide whether existing widget-related regression entries should be updated in light of the new capability.

## Out of scope

- Real cross-process integration between the two sample repos. The walkthrough validates the PM agent's cross-repo **coordination**, not a distributed system's wire protocol — acceptance tests on the API side mock the persistence port.
- Pagination, sorting, or additional filter predicates (status, tags, created-after). The minimum viable filter exercises cross-repo task decomposition without bloating scope.
- Authentication or rate limiting. The sample repos have none; the broader orchestrator architecture leaves those to real deployments.
- Any change to `.specfuse/verification.yml` or `.specfuse/templates.yaml` on either repo. Those were settled by the pre-walkthrough setup and the `templates.yaml` PR on `orchestrator-api-sample`.
