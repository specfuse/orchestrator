## Description

Feature used by the Phase 3 walkthrough (WU 3.6) to exercise the v1 QA agent end-to-end on a **happy-path, single-repo shape**. Adds a lightweight `GET /widgets/count` endpoint to `Bontyyy/orchestrator-api-sample` that returns the aggregate widget count without the widget payloads, surfaced as a narrow AC-1-only feature so that qa-authoring produces a single-test plan and qa-execution passes first-try against the merged implementation.

The feature is deliberately the **lowest-difficulty realistic shape** the QA pipeline can exercise end-to-end: one AC, one behavior, one implementation task, no regression cycle, no curation candidates. Its purpose inside the orchestrator is not its product value — it is to produce a signal on whether the v1 QA agent can take a validated feature spec through qa-authoring → qa-execution → feature close with zero human intervention beyond the walkthrough operator's orchestration.

See the walkthrough log at [`docs/walkthroughs/phase-3/feature-1-log.md`](../docs/walkthroughs/phase-3/feature-1-log.md) (created during the walkthrough) for the full narrative.

## Scope

- A new `GET /widgets/count` endpoint on the API component, returning `{"count": N}` where `N` is a non-negative integer equal to the number of widgets currently persisted.
- QA: one authored test plan covering AC-1, one execution run.

## Out of scope

- Filtering by widget status, owner, creation date, or any other criterion — AC-1 covers the unfiltered count only.
- Caching of the count across requests.
- Returning any widget fields beyond the count.
- Any change to persistence layer beyond invoking the existing widget-count capability (if the persistence layer exposes one) or iterating the in-memory collection — the feature is API-layer only.
