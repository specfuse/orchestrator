## Description

Feature used by the Phase 2 walkthrough (WU 2.7) to exercise the PM agent v1 on the **edge case shape: plan-review re-ingest stress**. The acceptance criterion from the implementation plan:

> **Plan-review re-ingest stress**: human edits the plan non-trivially (adds a task, retargets a dependency, tightens a work unit prompt); skill re-ingests faithfully.

The feature is a widget bulk-operations capability: a bulk import endpoint and a CSV export endpoint. Its intrinsic product value is secondary — the shape was chosen because it produces an initial task graph where several decisions (repo assignment, dependency ordering, scope of each implementation task) are defensibly-but-not-uniquely correct, giving the human room during `plan_review` to retarget, add, and tighten without the edits being artificial or contrived.

The edits the human will make during the walkthrough — retarget a dependency across repos, add a task, introduce then resolve a cycle, tighten a work unit prompt — exercise the plan-review skill's re-ingest path (cycle check, orphan check, assigned-repo sanity, task-ID uniqueness, schema round-trip, and correlation-ID immutability) on realistic human decisions rather than on synthetic test inputs.

See the walkthrough log at [`docs/walkthroughs/phase-2/feature-2-log.md`](../docs/walkthroughs/phase-2/feature-2-log.md) for the full narrative: skill invocations, per-edit re-ingest observations, any malformation-induced escalations, friction surfaced, and outcome.

## Behaviors

Each heading is one user-facing or system-facing behavior the feature delivers. The task-decomposition skill consumes them as the capability list (§Step 2).

### Behavior 1 — Bulk widget import

The API component exposes `POST /widgets/bulk` accepting a JSON array of widget payloads and persisting each via the existing creation path. On partial failure (some entries invalid, others valid), the endpoint returns `207 Multi-Status` with a per-entry success/error report; no entries are persisted if the request body itself fails top-level validation (non-array, empty array, or exceeds a 1000-entry ceiling).

### Behavior 2 — CSV export of current widget inventory

The API component exposes `GET /widgets/export.csv` streaming every stored widget as CSV with RFC 4180 quoting. Columns: `id,name,sku,quantity` in that order, header row included. The handler reads from the persistence component's existing list operation; no new persistence method is introduced for this behavior.

## Scope

- API repo: new `POST /widgets/bulk` endpoint with per-entry validation and aggregate reporting; new `GET /widgets/export.csv` streaming endpoint.
- Persistence repo: a new port method optimized for bulk insertion (atomic-ish addition of multiple widgets in one call) to back Behavior 1 efficiently; no new method for Behavior 2 (the existing `list_all_async` suffices).
- QA: authored test plan covering both behaviors including bulk-partial-failure semantics and CSV escaping edge cases; execution against the implementation; curation pass on regression entries.

## Out of scope

- Background / queued import. Bulk is synchronous, bounded by the 1000-entry ceiling.
- Import validation rules beyond the per-entry rules that already apply to single-widget creation (blank name, blank sku, negative quantity, and — per FEAT-2026-0001 — quantity above 10,000).
- Streaming import (chunked transfer). The request body is read fully before processing.
- CSV compression, pagination, or `Content-Disposition: attachment` headers beyond the minimum needed for a browser to render the stream as a download.
- Changes to `.specfuse/verification.yml` or `.specfuse/templates.yaml` on either repo.
