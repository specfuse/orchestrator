## Description

Feature used by the Phase 3 walkthrough (WU 3.6) to exercise the v1 QA agent's **regression-cycle path** end-to-end on a single-repo shape. Adds pagination to the `GET /widgets` listing endpoint on `Bontyyy/orchestrator-api-sample`, with a structured over-limit rejection (`page_size > 500` → HTTP 400 with error code `page_size_over_limit`) as the edge-case AC most plausible-to-miss on a first-pass implementation.

The feature is deliberately shaped so that a naive first-pass implementation can satisfy AC-1 and AC-2 (default and in-range slices) while missing AC-3 (structured error code on over-limit) — producing a qa-execution failure that drives the qa-regression skill through its full first-failure → inbox artifact → fix task → resolution loop. If the component agent happens to implement AC-3 correctly on the first pass, the walkthrough pivots to the backup plan (qa-curation stress on a seeded fixture corpus, documented in the walkthrough log).

See the walkthrough log at [`docs/walkthroughs/phase-3/feature-2-log.md`](../docs/walkthroughs/phase-3/feature-2-log.md) (created during the walkthrough) for the full narrative.

## Scope

- Paginate `GET /widgets` with a `page_size` query parameter: default 50 when absent, up to N for `1 ≤ N ≤ 500`, HTTP 400 with `error.code = page_size_over_limit` when `N > 500`.
- QA: one authored test plan covering AC-1 / AC-2 / AC-3 (three tests expected under the default cardinality convention), at least one execution run; the second execution run — against the fix commit — is expected on the regression-cycle primary path.

## Out of scope

- Cursor- or offset-based navigation beyond `page_size`.
- Sorting options.
- Filtering by any widget field.
- Behavior for non-integer or negative `page_size` values — the existing API-layer request-validation middleware on `orchestrator-api-sample` handles input-shape rejection; this feature is about the range-check semantics only.
