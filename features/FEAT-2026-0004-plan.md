## Task T01 — implementation, Bontyyy/orchestrator-persistence-sample

### Work unit prompt

Add an async method `list_by_quantity_above_async(threshold: int) -> list[Widget]` to the `WidgetRepository` Protocol in `src/persistence_sample/interfaces/widget_repository.py` and implement it in `InMemoryWidgetRepository` under `src/persistence_sample/adapters/in_memory_widget_repository.py`. The method returns every stored widget whose `quantity` is strictly greater than `threshold`, preserving the insertion order that `list_all_async` already provides. A threshold of `0` returns every widget with a positive quantity. The existing `asyncio.Lock` on the adapter is sufficient; no additional synchronization needed.

Acceptance:
- The new method exists on both the Protocol and the in-memory adapter, signatures match, fully async.
- Three new pytest cases at minimum: (a) returns widgets strictly above threshold, (b) threshold=0 filters out zero-quantity widgets, (c) empty repo returns `[]`.
- Coverage stays at 100% (threshold in `pyproject.toml` is 90%; the repo is currently 100%).
- `mypy src` passes strict.

Out of scope: changes to the API repo, changes to `.specfuse/verification.yml` or `.specfuse/templates.yaml`, changes to the `Widget` domain object.

Verification commands on this task: `pytest tests`, `pytest --cov=persistence_sample --cov-fail-under=90 tests`, `mypy src`, `ruff check src tests`.

Escalation: if the Protocol signature has materially changed since feature drafting (e.g. `list_all_async` was removed or renamed), escalate `spec_level_blocker` — the task's shape assumes the existing Protocol.

## Task T02 — implementation, Bontyyy/orchestrator-api-sample

### Work unit prompt

Add a `GET /widgets?min_quantity=N` endpoint to the existing `WidgetsController` (or the equivalent controller the sample repo surfaces widgets through). The endpoint returns a JSON array of widgets whose `quantity` is strictly greater than `N`. `N` is a non-negative integer; blank or missing `min_quantity` defaults to `0`. On non-integer or negative `min_quantity`, return `400 BadRequest` via the existing `ValidationException` pattern used by other widget endpoints.

Since this walkthrough validates cross-repo **coordination** and not a distributed wire protocol, the handler delegates to the persistence port through the repo's existing `IWidgetRepository` abstraction — **mock the `list_by_quantity_above_async` method (or its .NET-equivalent shape) in tests**; do not attempt a cross-process call to the Python persistence repo.

Acceptance:
- Endpoint exists, routed correctly, returns `200 OK` with JSON array on valid input.
- `400 BadRequest` on non-integer or negative `min_quantity`.
- Missing `min_quantity` defaults to `0` and returns all positive-quantity widgets.
- Service-level tests cover the three branches; controller-level tests cover the HTTP status mapping. Controller-level tests go into the existing `WidgetsControllerTests.cs` (which exists on main — verify at draft time per issue-drafting skill).
- Coverage stays ≥ 0.90.

Out of scope: changes to the persistence repo (it is handled by T01); changes to `.specfuse/verification.yml`; changes to authentication, rate limiting, or pagination.

Verification commands on this task: `dotnet test`, `dotnet test --collect:"XPlat Code Coverage"`, `dotnet format --verify-no-changes`.
