## Description

Feature used by the Phase 1 walkthrough (WU 1.5) in the orchestrator to exercise the v1 component agent configuration end-to-end on the sample component repository. Adds a maximum-quantity validation rule (`quantity > 10_000` → `ValidationException`) to the widget creation path in `Bontyyy/orchestrator-api-sample`, mirroring the three existing validation rules on `WidgetService.CreateAsync`.

The feature is deliberately the **lowest-difficulty realistic shape** the sample repo offers: a pattern-mirror validation rule, touching two files, with no design judgment required. Its purpose inside the orchestrator is not its product value (the sample repo has no product) — it is to produce a signal on whether the v1 component agent can take a well-formed work unit issue from `ready` through to a mergeable PR with zero human intervention. This is the explicit Task A criterion of WU 1.5.

See the walkthrough log at [`docs/walkthroughs/phase-1/task-A-log.md`](../docs/walkthroughs/phase-1/task-A-log.md) for the full narrative: pre-setup, issue content, agent actions, verification results, friction surfaced, and outcome.

## Scope

- Enforce `quantity <= 10_000` on widget creation in the Application-layer service, using the existing `ValidationException(field, reason)` convention with `field = "quantity"` and `reason = "must be at most 10000"`.
- Add unit-test coverage for the new rejection branch and an adjacent allowed-at-boundary case.

## Out of scope

- Any change to `WidgetsController`, `Widget` record, `IWidgetRepository`, or persistence layer — the new rule is enforced entirely in `WidgetService`, which already centralizes input validation.
- Introducing a configurable maximum. The ceiling is a literal `10_000` for this walkthrough, matching the shape of the existing `quantity < 0` rule. A parameterized maximum would be a separate feature.
- Any re-examination of the other validation rules already on `CreateAsync`. Scope is addition-only.
- Changes to `.specfuse/verification.yml` or CI — those were settled by the pre-walkthrough security-scan PR on the sample repo and are not this feature's concern.
