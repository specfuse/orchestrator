## Description

Feature used by the Phase 1 walkthrough (WU 1.5) in the orchestrator to exercise the v1 component agent configuration on a **moderate-difficulty** task. Adds a `DELETE /widgets/{id}` endpoint to `Bontyyy/orchestrator-api-sample` — a new HTTP verb on the existing widget resource, requiring a new repository port method, a new service method, a new controller action, persistence-layer implementation, and service-level unit tests.

The feature is the **moderate-shape walkthrough task** (WU 1.5 acceptance criterion 2): spans multiple files, requires genuine design judgment within tight constraints, and may attract one round of human PR review feedback. The bounded design question is the classic REST tradeoff — **idempotent `DELETE` (204 regardless of pre-existence)** versus **strict `DELETE` (404 when the id is unknown)**. The issue body pre-selects idempotent semantics and asks the agent to justify the choice in the PR description; the reviewer is free to push back, making this a realistic review-loop exercise.

Its purpose inside the orchestrator is to test whether the v1 component agent holds up when the task stretches beyond the single-file pattern-mirror shape of FEAT-2026-0001: the port + adapter pattern, the choice of return-signal on the repo method, the service-level behavior on missing ids, and the controller mapping from service result to HTTP response. See the walkthrough log at [`docs/walkthroughs/phase-1/task-B-log.md`](../docs/walkthroughs/phase-1/task-B-log.md) for the full narrative.

## Scope

- Add `Task DeleteAsync(string id, CancellationToken)` to `IWidgetRepository`, implement in `InMemoryWidgetRepository` using `ConcurrentDictionary.TryRemove`.
- Add `Task DeleteAsync(string id, CancellationToken)` to `WidgetService`, mirroring the existing `GetByIdAsync` input-validation shape (blank-id check → `ValidationException`).
- Add `[HttpDelete("{id}")]` action to `WidgetsController` returning `204 NoContent`, with the blank-id `ValidationException` surfaced as `400 validation_failed` (same convention as `POST`).
- Idempotent semantics: deleting a non-existent id returns `204 NoContent`, not `404`. Chosen to favor REST idempotence and reduce information leakage on arbitrary id probes.
- Add service-level unit tests on `WidgetService.DeleteAsync` (success, idempotent unknown, blank-id rejection) and controller-level tests on `WidgetsController.Delete` matching the shape of the existing `WidgetsControllerTests` coverage of `Create` and `GetById`. Controller tests were added by a PM amendment mid-task after the initial issue body mistakenly excluded them (see Task B log).

## Out of scope

- Changing any behavior of `CreateAsync` or `GetByIdAsync`.
- Introducing soft-delete, audit logging, or deletion authorization — the widget resource has no auth in this sample repo, and adding either is categorically outside a DELETE endpoint addition.
- Integration tests that run the full API host — per-task tests are service-level and in-memory controller-level, matching the existing convention in this repo.
- Changes to `.specfuse/verification.yml`, CI, branch protection, labels — the repo is bootstrapped.
- Any change to `Widget` domain record or to validation exceptions.
