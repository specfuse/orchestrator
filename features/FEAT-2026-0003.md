## Description

Feature used by the Phase 1 walkthrough (WU 1.5) in the orchestrator to exercise the v1 component agent configuration on an **edge-case** task — specifically, a task whose issue body carries a genuine, product-level specification ambiguity the agent cannot resolve by reading the component repository. The feature extends `POST /widgets` with a uniqueness rule on the `Name` field so two widgets cannot coexist with the same name.

The feature is the **edge-case walkthrough task** (WU 1.5 acceptance criterion 3): designed to exercise an escalation path the walkthrough has not yet touched. After FEAT-2026-0001 (trivial happy path) and FEAT-2026-0002 (moderate happy path), the most-useful-next-test is a `spec_level_blocker` escalation. The issue body for T01 is deliberately under-specified on three product-level axes (case sensitivity of the uniqueness comparison, error-response shape, whitespace handling) — each is a question no amount of code reading can answer, because the sample repo has no precedent for any of them.

The expected outcome of this feature is **not** a merged PR. The expected outcome is a well-formed escalation: a filed spec issue against the orchestrator repo (used as the specs-repo surrogate for walkthrough features, since the sample repo has no upstream specs project — see FEAT-2026-0002.md for the same precedent), a `human-escalation.md` inbox file, the issue label rotated to `state:blocked-spec`, and three events appended to the feature event log (`spec_issue_raised`, `task_blocked`, `human_escalation`). If the component agent pushes through the ambiguity and ships an arbitrary convention, that is a finding of the v1 configuration, not a pass.

See the walkthrough log at [`docs/walkthroughs/phase-1/task-C-log.md`](../docs/walkthroughs/phase-1/task-C-log.md) for the full narrative.

## Scope

- Add a uniqueness check on `Widget.Name` at the service layer in `WidgetService.CreateAsync`.
- Extend `IWidgetRepository` with a read method sufficient to support the uniqueness check if one is not already available.
- Surface rejection through `WidgetsController.Create` consistently with the feature's chosen error contract.
- Add unit tests in `WidgetServiceTests.cs` covering duplicate rejection, non-duplicate acceptance, and re-creation of a name freed by a prior `DELETE`.

## Out of scope

- Any change to `GetByIdAsync`, `DeleteAsync`, the quantity-ceiling rule on `CreateAsync`, or the blank-id validation.
- Soft-delete semantics, audit logging, or multi-tenant namespacing of names — the widget resource has no tenancy in this sample repo.
- Integration tests that run the full API host — per-task tests stay service-level + in-memory controller-level, matching FEAT-2026-0002's precedent.
- Changes to `.specfuse/verification.yml`, CI, branch protection, labels.
- Any decision the issue body does not explicitly resolve. If the issue body is ambiguous on a product-level point, the agent escalates rather than infers.
