# PM agent — plan-review UX skill (v1.0)

## Purpose

This skill is the PM agent's surface for the `plan_review` stage of a feature's lifecycle. It takes the task graph that [`../task-decomposition/SKILL.md`](../task-decomposition/SKILL.md) wrote into the feature frontmatter, materializes it as a **human-editable markdown document** at a predictable path, and provides the re-ingest procedure that writes human edits back into the feature frontmatter.

The plan-review document is the human's primary decision point on a feature. Its quality is a UX-quality decision that compounds across every feature: a format that is difficult to diff, edit, or parse forces friction into every planning session.

## Scope

In scope:

- **Phase A — Plan-file emission.** Once, after task decomposition completes. Writes the plan-review file, flips the feature state `planning → plan_review`, and emits `plan_ready`.
- **Phase B — Re-ingest.** Each time the plan file is edited by the human (typically many times during `plan_review`). Re-reads the plan file, re-validates, and re-writes the updated task graph into the feature frontmatter. Silent — no event per re-ingest.

Out of scope (and therefore not touched by this skill):

- The `plan_review → generating` transition. The human owns it per [`/shared/rules/state-vocabulary.md`](../../../../shared/rules/state-vocabulary.md) and architecture §6.3.
- The `plan_approved` event emission. The human emits it through whatever signaling mechanism the deployment uses (see §"Approval signaling" below). This skill does not emit `plan_approved`.
- The **trigger-detection loop** that invokes Phase B on plan-file edits. That is a polling-loop / commit-hook / CLI concern, outside the skill. The skill provides the re-ingest subroutine; something else decides when to call it.
- Drafting GitHub issue bodies. That is [`../issue-drafting/SKILL.md`](../issue-drafting/SKILL.md) (WU 2.4), triggered by `plan_approved` and the transition to `generating`.
- Modifying the task-decomposition skill's output shape. The task graph's schema is the upstream contract and this skill consumes it unchanged.

## Inputs

- The feature registry file at `/features/<correlation_id>.md` — its frontmatter is the canonical source for Phase A emission, and the canonical sink for Phase B re-ingest.
- The plan-review file at `/features/<correlation_id>-plan.md` — the human-edited surface this skill reads during Phase B.
- [`/shared/schemas/feature-frontmatter.schema.json`](../../../../shared/schemas/feature-frontmatter.schema.json) — the validation contract both phases enforce.
- [`/shared/schemas/event.schema.json`](../../../../shared/schemas/event.schema.json) — the contract for the `plan_ready` event emitted in Phase A.
- This skill and [`../../CLAUDE.md`](../../CLAUDE.md) — re-read on every invocation per [`/shared/rules/role-switch-hygiene.md`](../../../../shared/rules/role-switch-hygiene.md).

## Outputs

**Phase A:**

- `/features/<correlation_id>-plan.md` — the plan-review file, newly written.
- Update to `/features/<correlation_id>.md` frontmatter: `state` flipped from `planning` to `plan_review`. The `task_graph` is unchanged on emission — it is what 2.2 wrote.
- One `plan_ready` event appended to `/events/<correlation_id>.jsonl`.

**Phase B:**

- Update to `/features/<correlation_id>.md` frontmatter: `task_graph`, `involved_repos`, and `autonomy_default` synchronized with the plan file. `state` remains `plan_review`. `correlation_id` never changes.
- No event. Re-ingest is silent; auditability comes from git history on both files.

## Plan-file format

The plan-review file has three parts, in order: a YAML frontmatter header, one canonical `## Task graph` fenced YAML block (the source of truth for structural fields), and one `### Task TNN` prose section per task (the source of truth for work unit prompt drafts).

```markdown
---
feature_correlation_id: FEAT-YYYY-NNNN
plan_pass: 1
---

# Plan review — FEAT-YYYY-NNNN

<one-paragraph reminder of what this file is, generated by the skill>

## Task graph

```yaml
involved_repos:
  - owner/repo-a
  - owner/repo-b
autonomy_default: review
task_graph:
  - id: T01
    type: implementation
    depends_on: []
    assigned_repo: owner/repo-a
  - id: T02
    type: qa_authoring
    depends_on: []
    assigned_repo: owner/repo-a
  - id: T03
    type: qa_execution
    depends_on: [T01, T02]
    assigned_repo: owner/repo-a
    autonomy: review
```

## Task T01 — implementation, owner/repo-a

### Work unit prompt

<free-form prose that the human and the PM agent co-author. This becomes the
working draft that the issue-drafting skill (WU 2.4) will consume into a
work-unit-issue.md-compliant body.>

## Task T02 — qa_authoring, owner/repo-a

### Work unit prompt

<prose>

## Task T03 — qa_execution, owner/repo-a

### Work unit prompt

<prose>
```

### Two edit surfaces

The file has two kinds of editable content, with a clear split:

- **Structural fields** (`task_graph`, `involved_repos`, `autonomy_default`): edited inside the single fenced YAML block under `## Task graph`. These are the fields the feature-frontmatter schema governs.
- **Work unit prompts**: edited inside each `### Work unit prompt` prose section. Free-form; the schema does not constrain them. They live in the plan file only and are consumed by [`../issue-drafting/SKILL.md`](../issue-drafting/SKILL.md) (WU 2.4) at drafting time.

The `## Task TNN — <type>, <repo>` heading line is a human-readable rendering of the task's structural fields. It is **regenerated from the YAML block on every emission and every re-ingest**; edits to the heading line are ignored. If the human wants to change a task's type or repo, they edit the YAML block; the heading reflows on the next round-trip.

### Why this format

- **Diffable.** Edits to structural fields show as small YAML diffs (e.g. one `depends_on:` line changes). Edits to prompts show as small prose diffs inside the relevant `### Task TNN` section. A git diff of a realistic edit is readable line-by-line, not a wholesale replacement.
- **Parseable.** The YAML block is a single fenced region with a known header; extracting it for re-ingest is a small deterministic operation. No table parsing, no prose-to-structure inference.
- **Two-surface split.** Structural edits and prompt edits do not collide. The YAML is the machine contract; the prose is the human-to-agent contract for drafting.
- **No duplicate source of truth.** The feature frontmatter's `task_graph` is always the canonical live state during `plan_review`; the plan file's YAML block is overwritten back into the frontmatter on every re-ingest, keeping them in sync. When the next phase (`generating`) begins, the plan file is historical and the frontmatter alone governs.

## Phase A — Plan-file emission

Invoked once, after task decomposition has written a `task_graph` into the feature frontmatter and emitted `task_graph_drafted`.

### Pre-conditions

The skill confirms, by direct read:

1. `/features/<correlation_id>.md` exists and its frontmatter has `state: planning`.
2. The frontmatter's `task_graph` is non-empty (decomposition has run).
3. No plan file currently exists at `/features/<correlation_id>-plan.md`. If it does, the skill is running an **emission pass ≥ 2** (re-decomposition has occurred and this is a fresh emission); the existing plan file is overwritten, the `plan_pass` counter is incremented, and the overwrite is the expected behavior.

### Procedure

1. State intent per [`verify-before-report.md`](../../../../shared/rules/verify-before-report.md) §1: "I will emit the plan-review file for `<correlation_id>`."
2. Construct the plan-file content:
   - Frontmatter `feature_correlation_id: <id>`, `plan_pass: <n>` (1 for initial emission, else the previous pass + 1).
   - Copy `involved_repos`, `autonomy_default`, and `task_graph` verbatim into the `## Task graph` YAML block.
   - Generate a `### Task TNN — <type>, <assigned_repo>` section per task, in `task_graph` order. The `### Work unit prompt` subsection starts with a placeholder line (`<draft the work unit prompt for this task>`) on initial emission. On emission pass ≥ 2, preserve any existing prose from the previous plan file for tasks whose `id` is unchanged — re-emission is about structural refresh, not prose reset.
3. Write the plan file to `/features/<correlation_id>-plan.md`. Re-read it back; confirm the YAML block parses and the task ordering matches the source frontmatter.
4. Update the feature frontmatter: flip `state` from `planning` to `plan_review`. Touch no other field.
5. Construct the `plan_ready` event:
   - `timestamp`: ISO-8601 now.
   - `correlation_id`: the feature-level ID.
   - `event_type`: `plan_ready`.
   - `source`: `pm`.
   - `source_version`: from `scripts/read-agent-version.sh pm` at emission time.
   - `payload`: `{"plan_file_path": "features/<id>-plan.md", "task_count": <n>, "plan_pass": <pass>}`.
6. Pipe the event through `scripts/validate-event.py`; require exit 0; append to `/events/<correlation_id>.jsonl`. Re-read the appended line.

### Phase A exit criteria

`plan_ready` is in the event log. Feature state is `plan_review`. Plan file exists on disk. The skill **does not loop**: control returns to whatever invoked it, and Phase B waits for a separate external trigger. The skill is not a daemon.

## Phase B — Re-ingest

Invoked each time the human edits the plan file. The trigger-detection mechanism is outside this skill's scope (see §"Approval signaling" and §"The trigger-detection loop is external"). When invoked, the skill performs the following procedure **from scratch** — no caching of prior reads, no incremental diff, no assumption about which fields changed.

### Procedure

1. State intent: "I will re-ingest the plan-review file for `<correlation_id>`."
2. Read `/features/<correlation_id>-plan.md` in full.
3. Parse the top-level frontmatter. Confirm `feature_correlation_id` matches the target feature file. If it does not, escalate `spec_level_blocker` with reason "plan file's `feature_correlation_id` does not match feature registry".
4. Extract the `## Task graph` fenced YAML block. It must be the only fenced YAML block under that heading. Parse it.
5. Build the candidate feature-frontmatter object by merging:
   - `correlation_id`: from the existing feature frontmatter (never taken from the plan file body — the plan file's frontmatter is metadata about the plan file, not about the feature).
   - `state`: forced to `plan_review` (re-ingest does not change feature state).
   - `involved_repos`, `autonomy_default`, `task_graph`: from the plan file's YAML block.
6. Validate the candidate against [`feature-frontmatter.schema.json`](../../../../shared/schemas/feature-frontmatter.schema.json). Any schema failure → escalate `spec_level_blocker` with reason "re-ingest schema validation failed: <error path and message>".
7. Structural validation beyond the schema:
   - **Cycle check** on `task_graph.depends_on` edges (topological sort must sort all tasks).
   - **Orphan check**: every `TNN` in any `depends_on` array must match a task `id`.
   - **Assigned-repo sanity**: every task's `assigned_repo` ∈ `involved_repos`.
   - **Task-ID uniqueness**: no duplicate `id`.
   - **Correlation-ID immutability**: the plan file's YAML block must not attempt to change the feature correlation ID (step 5 already enforces this, but check explicitly for the case where someone adds a `correlation_id` field inside the YAML block).
   Any failure → escalate `spec_level_blocker`.
8. Write the validated object to `/features/<correlation_id>.md` frontmatter. Preserve the prose body unchanged.
9. Re-read the feature file back. Confirm the frontmatter round-trips.
10. Do **not** emit an event. The plan file's git history and the feature file's git history are the audit trail; adding a `plan_reingested` event per edit would add noise without new information.

### What re-ingest does NOT enforce

Re-ingest enforces **structural integrity and schema conformance**, not the full set of heuristics the task-decomposition skill applied at decomposition time. In particular, re-ingest does not check:

- QA task pairing (every `qa_execution` has a matching `qa_authoring` on the same repo). The human may deliberately decouple them or drop a QA task entirely; the skill does not second-guess.
- Target-repo inference rules. If the human retargets a task to a repo whose inference-rule mapping would have assigned a different repo at decomposition time, the human's choice wins. They have context the skill does not.
- Autonomy override heuristics (qa_execution never auto, sensitive-path → supervised). The human can weaken or strengthen autonomy on any task.

The human is the authority during `plan_review`; the skill validates that their edits are **structurally coherent**, not that they match the agent's original policy. This is by design — the `plan_review` stage exists precisely so the human can override heuristic decisions.

### Phase B exit criteria

Either: the candidate validated and the feature frontmatter has been updated, and control returns to the caller. Or: validation failed and the skill has escalated `spec_level_blocker` with a specific reason; the feature state remains `plan_review`; the plan file is untouched; the feature frontmatter is untouched.

Re-ingest is idempotent. Running it twice in a row on an unchanged plan file produces no-op writes (same content), though the skill still performs the full read-parse-validate cycle each time — there is no "skip because nothing changed" shortcut.

## Approval signaling

The `plan_review → generating` transition belongs to the human. The human signals approval; the skill does not detect approval by reading the plan file.

The recommended convention for the initial deployment is a **human-escalation-style inbox file** at `/inbox/plan-approved/<correlation_id>.md`, written by the human (or a helper script acting on the human's explicit request). A polling loop detects the file, emits a `plan_approved` event (with `source: human`), and moves the file to an `archive/` subdirectory. This mechanism is consistent with [`escalation-protocol.md`](../../../../shared/rules/escalation-protocol.md) and does not require GitHub labels (which would require a feature-level tracking issue that does not otherwise exist during `plan_review`).

Alternative signaling mechanisms (GitHub label, commit-message convention, manual event append) are acceptable; the skill is agnostic to the specific mechanism and does not read any signal file. The skill's responsibility ends at writing the feature frontmatter correctly; what the human does next is the polling loop's concern.

### The trigger-detection loop is external

The Phase B re-ingest runs in response to plan-file edits. Detecting that edits have occurred is not this skill's responsibility. Candidate mechanisms (any of which are acceptable, depending on deployment):

- A polling loop that `git diff`s the plan file against its last-ingested revision.
- A git post-commit hook that invokes the skill.
- A manual CLI (`scripts/reingest-plan.sh FEAT-YYYY-NNNN`) the human runs after each edit.
- A file-watcher (inotify / FSEvents) during local development.

The skill is a function the loop calls; it is not a daemon. Choosing a trigger mechanism is a deployment decision and lives in the orchestrator's runtime scripts, not in this skill.

## Verification

Before returning from Phase A:

- The plan file exists at the declared path.
- The YAML block inside the plan file parses and matches the source frontmatter.
- The feature frontmatter's `state` is `plan_review`.
- The `plan_ready` event passed `scripts/validate-event.py` (exit 0) and is the last line of the feature's event log.
- `source_version` on the event was produced by `scripts/read-agent-version.sh pm` at emission.
- No path written is in [`never-touch.md`](../../../../shared/rules/never-touch.md).

Before returning from Phase B:

- The feature frontmatter re-validates against [`feature-frontmatter.schema.json`](../../../../shared/schemas/feature-frontmatter.schema.json).
- Topological sort of `depends_on` succeeds; no cycles.
- No orphan `TNN` references.
- Every `assigned_repo` ∈ `involved_repos`.
- No duplicate task IDs.
- The feature file's prose body is byte-identical to its pre-ingest content. The skill touches the frontmatter only.

Failure handling per [`verify-before-report.md`](../../../../shared/rules/verify-before-report.md) §3: locally-correctable failures retry; three consecutive failures → `spinning_detected`; fundamentally-blocked → `spec_level_blocker`.

## Worked example

Feature `FEAT-2026-0051 — Widget search by tag`. The task-decomposition skill (2.2) ran first and produced a three-task graph on a single repo.

### Initial feature frontmatter (after 2.2, before 2.3 Phase A)

```yaml
---
correlation_id: FEAT-2026-0051
state: planning
involved_repos:
  - clabonte/api-sample
autonomy_default: review
task_graph:
  - id: T01
    type: implementation
    depends_on: []
    assigned_repo: clabonte/api-sample
  - id: T02
    type: qa_authoring
    depends_on: []
    assigned_repo: clabonte/api-sample
  - id: T03
    type: qa_execution
    depends_on: [T01, T02]
    assigned_repo: clabonte/api-sample
---
```

### Phase A — emission

The skill writes `/features/FEAT-2026-0051-plan.md`:

````markdown
---

---

Edit the YAML block under "## Task graph" to adjust structural fields
(dependencies, autonomy, repo assignments, involved repos). Edit the prose
under each "### Work unit prompt" section to co-author the prompt draft
that the issue-drafting skill will consume. The `## Task TNN …` heading
line is regenerated on every round-trip; edits to it are ignored.

## Task graph

```yaml
involved_repos:
  - clabonte/api-sample
autonomy_default: review
task_graph:
  - id: T01
    type: implementation
    depends_on: []
    assigned_repo: clabonte/api-sample
  - id: T02
    type: qa_authoring
    depends_on: []
    assigned_repo: clabonte/api-sample
  - id: T03
    type: qa_execution
    depends_on: [T01, T02]
    assigned_repo: clabonte/api-sample
```

## Task T01 — implementation, clabonte/api-sample

### Work unit prompt

<draft the work unit prompt for this task>

## Task T02 — qa_authoring, clabonte/api-sample

### Work unit prompt

<draft the work unit prompt for this task>

## Task T03 — qa_execution, clabonte/api-sample

### Work unit prompt

<draft the work unit prompt for this task>
````

The feature frontmatter's `state` flips to `plan_review`. A `plan_ready` event is appended:

```json
{
  "timestamp": "2026-04-22T19:00:00Z",
  "correlation_id": "FEAT-2026-0051",
  "event_type": "plan_ready",
  "source": "pm",
  "source_version": "1.0.0",
  "payload": {
    "plan_file_path": "features/FEAT-2026-0051-plan.md",
    "task_count": 3,
    "plan_pass": 1
  }
}
```

### Human edit (non-trivial)

The human, reviewing the plan, decides three things:

1. The search logic belongs in the persistence repo, not the API repo. T01 is retargeted to `clabonte/persistence-sample`, and `involved_repos` gains that repo.
2. The test plan (T02) should be authored after the implementation exists (the human wants to test against the real impl shape). T02's `depends_on` becomes `[T01]`.
3. T02's work unit prompt is tightened from a placeholder to a specific directive.

The human edits the plan file:

````markdown
## Task graph

```yaml
involved_repos:
  - clabonte/api-sample
  - clabonte/persistence-sample
autonomy_default: review
task_graph:
  - id: T01
    type: implementation
    depends_on: []
    assigned_repo: clabonte/persistence-sample
  - id: T02
    type: qa_authoring
    depends_on: [T01]
    assigned_repo: clabonte/api-sample
  - id: T03
    type: qa_execution
    depends_on: [T01, T02]
    assigned_repo: clabonte/api-sample
```

## Task T01 — implementation, clabonte/api-sample

### Work unit prompt

<draft the work unit prompt for this task>

## Task T02 — qa_authoring, clabonte/api-sample

### Work unit prompt

Author a test plan for the tag-search widget endpoint. Cover: (a) happy-path
lookup by single tag, (b) multi-tag intersection semantics, (c) empty-result
behavior, (d) tag normalization (case, whitespace). Read the implementation
in clabonte/persistence-sample's search port before finalizing assertions;
the behavior the tests assert must match what T01 delivers, not a
speculative contract.

## Task T03 — qa_execution, clabonte/api-sample

### Work unit prompt

<draft the work unit prompt for this task>
````

Note: the `## Task T01 — implementation, clabonte/api-sample` heading still says `api-sample`, unchanged by the human. That heading is a rendering; the next re-ingest will regenerate it from the YAML block (where T01 now says `persistence-sample`). The human does not need to re-write heading lines.

### Phase B — re-ingest

The skill, invoked by whatever external trigger detected the edit:

1. Reads the plan file. Frontmatter `feature_correlation_id` matches. ✓
2. Extracts the `## Task graph` YAML block.
3. Builds the candidate frontmatter:
   ```yaml
   correlation_id: FEAT-2026-0051  # from existing frontmatter
   state: plan_review              # forced
   involved_repos:
     - clabonte/api-sample
     - clabonte/persistence-sample
   autonomy_default: review
   task_graph:
     - id: T01
       type: implementation
       depends_on: []
       assigned_repo: clabonte/persistence-sample
     - id: T02
       type: qa_authoring
       depends_on: [T01]
       assigned_repo: clabonte/api-sample
     - id: T03
       type: qa_execution
       depends_on: [T01, T02]
       assigned_repo: clabonte/api-sample
   ```
4. Schema validation: passes. ✓
5. Cycle check: `T01 → T02 → T03`, topologically sorted. ✓
6. Orphan check: `{T01, T02}` ⊆ `{T01, T02, T03}`. ✓
7. Assigned-repo sanity: `{persistence-sample, api-sample, api-sample}` ⊆ `{api-sample, persistence-sample}`. ✓
8. Task-ID uniqueness: `{T01, T02, T03}` all distinct. ✓
9. Writes the frontmatter.
10. Re-reads; confirms.
11. **No event emitted.** Re-ingest is silent.

The feature frontmatter is now:

```yaml
---
correlation_id: FEAT-2026-0051
state: plan_review
involved_repos:
  - clabonte/api-sample
  - clabonte/persistence-sample
autonomy_default: review
task_graph:
  - id: T01
    type: implementation
    depends_on: []
    assigned_repo: clabonte/persistence-sample
  - id: T02
    type: qa_authoring
    depends_on: [T01]
    assigned_repo: clabonte/api-sample
  - id: T03
    type: qa_execution
    depends_on: [T01, T02]
    assigned_repo: clabonte/api-sample
---
```

The prose body of the feature file is unchanged. A `git diff` on the feature file shows three lines in the frontmatter — the added `involved_repos` entry, the retargeted `assigned_repo` on T01, and the `depends_on: [T01]` on T02 — and nothing else. The plan file's `git diff` shows the corresponding YAML edits plus the prose edit under T02's prompt section. Both diffs are readable line-by-line.

### Next steps (outside this skill)

The human, satisfied, signals approval — e.g. by writing `/inbox/plan-approved/FEAT-2026-0051.md`. A polling loop detects the file, emits `plan_approved` with `source: human`, and begins the `plan_review → generating` flip. None of that is the plan-review skill's concern.

## What this skill does not do

- It does **not** flip `plan_review → generating`. That transition is the human's, per architecture §6.3.
- It does **not** emit `plan_approved`. The human emits it (directly or via a helper).
- It does **not** detect plan-file edits. The trigger-detection loop is external; this skill is a function, not a daemon.
- It does **not** modify the feature file's prose body. It writes the frontmatter only.
- It does **not** emit events on re-ingest. Git history is the audit surface for per-edit changes.
- It does **not** re-apply task-decomposition heuristics (QA pairing, target-repo inference rules, autonomy override rules) on re-ingest. The human is the authority during `plan_review`; the skill validates structural integrity only.
- It does **not** draft GitHub issue bodies. That is [`../issue-drafting/SKILL.md`](../issue-drafting/SKILL.md) (WU 2.4), invoked after `plan_approved` and the state flip to `generating`.
