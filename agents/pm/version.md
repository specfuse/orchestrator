# PM agent version

Current version: **1.2.0**

## Changelog

- `1.2.0` — WU 2.10: task-decomposition skill clarifications and issue-drafting prose-narration clause. (a) `task-decomposition/SKILL.md` Step 4: added "Feature scope overrides" subsection — when the feature `## Scope` explicitly constrains `qa_authoring` cardinality (e.g. "one authored test plan covering both behaviors"), that constraint overrides the default per-behavior count; the default applies only when `## Scope` is silent on cardinality. Closes findings F1.9, F2.1. (b) `task-decomposition/SKILL.md` Step 5 rule 3: rewrote `qa_execution depends_on` to use a same-behavior gate rather than a whole-repo gate — `qa_execution` depends on the implementation tasks for the same behavior as its matched `qa_authoring`, not all implementation tasks on the repo; cross-repo impl tasks mocked at the test boundary are excluded; includes worked example showing 2-behavior same-repo feature. Closes findings F1.10, F2.2. (c) `task-decomposition/SKILL.md` `decomposition_pass` payload field: added explicit statement that the counter is not persisted in the feature frontmatter and is derived by counting prior `task_graph_drafted` events in the event log at emission time. Closes finding F1.11. (d) `task-decomposition/SKILL.md` Step 7: added orphan-check asymmetry note — adding a task risks orphans on the new task's own `depends_on`; removing a task risks orphans in any surviving task's `depends_on`. Folds positive observation F2.13 into skill text. (e) `issue-drafting/SKILL.md` Step 3: added upstream-task narration clause — for any task with non-empty `depends_on`, both the YAML `depends_on` field and a human-readable prose mention in `## Context` are required. Closes finding F1.12.
- `1.1.0` — WU 2.9: event schema additions — `feature_state_changed` and `human_escalation` per-type payload. (a) `feature_state_changed` emission points named explicitly in `CLAUDE.md` §"Output artifacts": `planning → plan_review` (`trigger: plan_ready`), `plan_review → generating` (`trigger: plan_approved`), `generating → in_progress` (`trigger: first_round_issues_opened`), `in_progress → done` (`trigger: all_tasks_done`). (b) New per-type payload schemas added at `shared/schemas/events/feature_state_changed.schema.json` (fields: `from_state`, `to_state`, `trigger`) and `shared/schemas/events/human_escalation.schema.json` (fields: `summary`, `reason` enum, optional `cycle_members` array). (c) F2.9 escalation-state tension resolved in §"Output artifacts": `human_escalation` event is the authoritative blocked signal; PM agent does NOT write to feature frontmatter `state` on escalation. Closes findings F1.1, F2.8, F2.9.
- `1.0.0` — WU 2.1: PM agent configuration rewritten to production v1 quality. `CLAUDE.md` restructured around the five Phase 2 skills (task decomposition, plan review, issue drafting, dependency recomputation, template coverage), with role-specific verification clauses pointing to each skill, an anti-patterns section codifying the PM-side failure modes from Phase 1 walkthroughs, explicit output-artifact definitions, and a preserved "Phase 2 specification inputs" list for any additional inherited specs that later Phase 2 WUs may append. The shared edit is authorized against the Phase 1 freeze by the retrospective's explicit carve-out for Finding 6. `agents/pm/README.md` added for cold-open navigation. `agents/pm/rules/` remains empty by design; no role-specific overrides are needed at v1. Closes WU 2.1 of Phase 2.
