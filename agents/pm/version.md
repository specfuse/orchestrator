# PM agent version

Current version: **1.4.0**

## Changelog

- `1.4.0` — WU 2.12: plan-review skill polish. (a) Structural-vs-prose edit distinction clarified in §"Two edit surfaces": structural edits (YAML-block changes: task additions/removals, `assigned_repo`, `depends_on`, `required_templates` changes) trigger Phase B re-ingest; prose edits (`### Work unit prompt` bodies) are the primary human authoring task and do not trigger re-ingest structurally. The "no edits during plan review" criterion in WU 2.7 acceptance criteria refers to structural edits only. Closes finding F1.13. (b) Stale-heading behavior documented explicitly in §"Two edit surfaces": heading lines are regenerated on Phase A emission only; Phase B leaves the plan file untouched; heading lines can therefore become stale between Phase A emissions when Phase B accepts YAML-block changes; the YAML block is canonical and is what consumers (human, issue-drafting skill) read from. Worked example note corrected to remove the incorrect "next re-ingest will regenerate" claim. Closes finding F2.12. (c) Human-authoring sequence during `plan_review` made explicit in new §"Human-authoring steps during `plan_review`" sub-section (placed after Phase A exit criteria): enumerates the 5-step sequence from Phase A emission through `required_templates` population (structural edit, triggers re-chain pre-flight) through prose drafting through approval signaling. First-time reader no longer needs to cross-reference Out-of-scope clauses in two other skills to reconstruct the sequence. Closes finding F1.2. Skill internal version bumped: `plan-review/SKILL.md` v1.1 → v1.2.
- `1.3.0` — WU 2.11: template-coverage skill contract hardening + plan-review Phase B chaining. (a) `template-coverage-check/SKILL.md` Out-of-scope clause rephrased as imperative prohibition: "DO NOT populate `required_templates` even if absent; escalate `spec_level_blocker` rather than infer the field." New Step 2.5 (pre-flight check) added: if any task in the feature's task graph is missing the `required_templates` key entirely (field absent — distinct from explicit empty array `[]`), the skill immediately escalates `spec_level_blocker` with a `human_escalation` event + inbox file naming the tasks; `qa_execution`/`qa_curation` tasks that carry `required_templates: []` are not escalated. Closes finding F2.3. (b) `plan-review/SKILL.md` Phase B extended with §"Template-coverage re-chain" final step: after every successful structural validation, Phase B unconditionally re-invokes the full template-coverage-check procedure (pre-flight + coverage walk) against the post-edit graph. On pass, emits a fresh `template_coverage_checked` event (validated through `scripts/validate-event.py` before append). On pre-flight failure or coverage gap, escalates `spec_level_blocker` via the same path as primary coverage-check invocations. Phase B Outputs, Verification, exit-criteria, Scope, and "What this skill does not do" sections all updated to reflect the re-chain. Closes findings F2.5 and F2.7. (c) Event emission discipline: every Phase B success now leaves a `template_coverage_checked` event in the log (unconditional, not conditional on change detection), making coverage status at each re-ingest observable in the audit trail. Skill internal versions bumped: `template-coverage-check/SKILL.md` v1.0 → v1.1; `plan-review/SKILL.md` v1.0 → v1.1.
- `1.2.0` — WU 2.10: task-decomposition skill clarifications and issue-drafting prose-narration clause. (a) `task-decomposition/SKILL.md` Step 4: added "Feature scope overrides" subsection — when the feature `## Scope` explicitly constrains `qa_authoring` cardinality (e.g. "one authored test plan covering both behaviors"), that constraint overrides the default per-behavior count; the default applies only when `## Scope` is silent on cardinality. Closes findings F1.9, F2.1. (b) `task-decomposition/SKILL.md` Step 5 rule 3: rewrote `qa_execution depends_on` to use a same-behavior gate rather than a whole-repo gate — `qa_execution` depends on the implementation tasks for the same behavior as its matched `qa_authoring`, not all implementation tasks on the repo; cross-repo impl tasks mocked at the test boundary are excluded; includes worked example showing 2-behavior same-repo feature. Closes findings F1.10, F2.2. (c) `task-decomposition/SKILL.md` `decomposition_pass` payload field: added explicit statement that the counter is not persisted in the feature frontmatter and is derived by counting prior `task_graph_drafted` events in the event log at emission time. Closes finding F1.11. (d) `task-decomposition/SKILL.md` Step 7: added orphan-check asymmetry note — adding a task risks orphans on the new task's own `depends_on`; removing a task risks orphans in any surviving task's `depends_on`. Folds positive observation F2.13 into skill text. (e) `issue-drafting/SKILL.md` Step 3: added upstream-task narration clause — for any task with non-empty `depends_on`, both the YAML `depends_on` field and a human-readable prose mention in `## Context` are required. Closes finding F1.12.
