# QA agent version

Current version: **1.2.0**

## Changelog

- `1.2.0` — WU 3.3: `agents/qa/skills/qa-execution/SKILL.md` v1.0 authored as the second Phase 3 skill. The skill reads a plan written by qa-authoring, runs each test's `commands` against the assigned component repo, evaluates the `expected` predicate (agent-judgment at v1; Phase 4 machine-evaluable predicates), and emits either `qa_execution_completed` or `qa_execution_failed` — one event per `(task_correlation_id, commit_sha)` pair, with the pair enforced as the idempotence key before emission. Two new per-type payload schemas: `shared/schemas/events/qa_execution_completed.schema.json` (`{task_correlation_id, commit_sha, plan_path, test_count, total_duration_seconds?}`) and `shared/schemas/events/qa_execution_failed.schema.json` (adds `failed_tests[]` of `{test_id, first_signal, exit_code?, stderr_excerpt?}`, minItems 1). `qa_execution_completed` + `qa_execution_failed` added additively to the `event.schema.json` enum. **Decision — D3 carried through:** the `implementation_task_correlation_id` being regressed against is NOT carried in the failed payload; WU 3.4 (qa-regression) resolves it from the feature task graph (SOT = feature registry), because feature-scope overrides on `qa_authoring` cardinality (WU 2.10) make the test→implementation mapping not 1:1. **Q6 / Finding 8 disposition recorded in the skill's `## Finding 8 disposition` section and in the commit message: Finding 8 does NOT apply to qa-execution. Carry reaffirmed to Phase 5.** Rationale: qa-execution never invokes `--no-build`; its commands are black-box against the running system. Indirect risk (pre-started service built from stale source) mitigated by the skill's stateless-command discipline and provenance-escalation rule (ambiguous service provenance → `spec_level_blocker`). No component-agent verification skill edit; Phase 1 freeze preserved. No follow-on Phase 3 fix WU surfaced. Worked example = FEAT-2026-0061 (widgets export rate-limit, 3 tests, 3 runs exercising all-pass / regression / idempotent-skip) fixtured under `shared/schemas/examples/qa_execution_completed.json` + `qa_execution_failed.json`. `## Deferred integration` section names Phase 4 (machine-evaluable predicates: JSONPath / CEL / custom mini-DSL; Arazzo-backed commands; formalized cascade semantics via `depends_on_test` + `kind: cascaded` markers) and Phase 5 (generator-emitted skeletons, qa-execution largely unchanged) as the homes for the v1 stubs. No change to the frozen PM (v1.6.0) or component (v1.5.0) surfaces. Closes WU 3.3 of Phase 3.
- `1.1.0` — WU 3.2: `agents/qa/skills/qa-authoring/SKILL.md` v1.0 authored as the first Phase 3 skill. The skill turns a validated feature spec into an executable test plan file at `/product/test-plans/FEAT-YYYY-NNNN.md` and emits a `test_plan_authored` event on success. Stub contract: `shared/schemas/test-plan.schema.json` (with `schema_version: 1`, `feature_correlation_id`, and a `tests[]` array of `{test_id, covers, commands, expected}` entries) validates the plan file's YAML frontmatter; richer Arazzo-backed and generator-emitted-skeleton evolutions are deferred to Phase 4 and Phase 5 respectively, documented in the skill's `## Deferred integration` section as a concrete Phase 4/5 brief. Per-type event payload schema `shared/schemas/events/test_plan_authored.schema.json` lands alongside (`plan_path` + `test_count`), following the WU 2.6 minimalist precedent. `test_plan_authored` added to the `shared/schemas/event.schema.json` enum additively. Worked example (FEAT-2026-0060 — widgets export pagination, 2 AC → 2 tests) fixtured under `shared/schemas/examples/test-plan.json` and `shared/schemas/examples/test_plan_authored.json`. No change to the frozen PM (v1.6.0) or component (v1.5.0) surfaces. Closes WU 3.2 of Phase 3.
- `1.0.0` — WU 3.1: QA agent configuration rewritten to production v1 quality. `CLAUDE.md` restructured around the four Phase 3 skills (`qa-authoring`, `qa-execution`, `qa-regression`, `qa-curation`), with a new top-level `## Cross-task regression semantics` section codifying the Q4 invariant: the QA agent never writes labels or state to a task it does not own, and a qa-execution failure spawns a new implementation task via `/inbox/qa-regression/` rather than flipping the state of the task under test. Role-specific verification clauses reference each Phase 3 skill by name (the skill files themselves land in WUs 3.2–3.5). Role-specific escalation expanded to distinguish qa-task-level `spinning_detected` (authoring/curation failures) from implementation-task-level `spinning_detected` on qa-execution repeat failure per architecture §6.4. New `## Anti-patterns` section codifies the Q4 invariant and the idempotence disciplines (qa-execution keyed on `(task_correlation_id, commit_sha)`; qa-regression keyed on `(implementation_task_correlation_id, test_id)`) as hard stops. Role-switch hygiene inherited from `shared/rules/role-switch-hygiene.md` (landed in WU 2.1); no QA-specific override. Output artifacts section now enumerates the specific Phase 3 events (`test_plan_authored`, `qa_execution_completed`, `qa_execution_failed`, `qa_regression_filed`, `qa_regression_resolved`, `escalation_resolved`, `regression_suite_curated`) alongside the shared-lifecycle events (`task_started`, `task_completed`, `task_blocked`, `spec_issue_raised`, `human_escalation`). `agents/qa/README.md` added for cold-open navigation. `agents/qa/rules/` remains empty by design; no role-specific overrides are needed at v1. Closes WU 3.1 of Phase 3.
